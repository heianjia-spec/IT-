## 1. 文档编写原则

### 1.1 核心原则

| 原则 | 说明 |
|------|------|
| **清晰** | 语言简洁明了，避免歧义，开发人员能直接理解 |
| **完整** | 覆盖所有业务场景，包括正常流程和异常处理 |
| **可验证** | 需求可被测试验证，有明确的验收标准 |
| **一致性** | 术语、格式、风格保持一致 |
| **可追溯** | 变更有记录，版本有历史 |

### 1.2 编写要求

- **使用中文**：所有文档内容使用中文编写
- **避免模糊词汇**：如"大约"、"可能"、"尽量"、"适当"等
- **量化指标**：性能要求、数量限制等必须有具体数值
- **举例说明**：复杂逻辑通过示例帮助理解

## 2. Frontmatter 规范

每个需求文档必须包含 YAML Frontmatter 元数据：

```yaml
---
模块: "{{模块名称}}"
状态: "待评审"
优先级: "P1"
负责人: "{{产品经理姓名}}"
创建人: "{{创建人姓名}}"
产品: "{{lmq 或 wuss}}"
迭代: ""
TAPD_ID: ""

创建日期: "{{YYYY-MM-DD}}"
更新日期: "2026-03-24"
版本号: "V1.0"

依赖: []
关联接口: []
---
```

### 2.1 字段说明

| 字段 | 必填 | 说明 | 可选值 |
|------|------|------|--------|
| 模块 | 是 | 功能模块名称 | 如：字典管理、用户管理 |
| 状态 | 是 | 文档当前状态 | 待评审/待开发/开发中/待测试/测试中/已完成/暂停 |
| 优先级 | 是 | 需求优先级 | P0(紧急)/P1(高)/P2(中)/P3(低) |
| 负责人 | 是 | 文档编写人 | wuss / lmq / wl |
| 创建人 | 是 | 文档创建人 | wuss / lmq / wl |
| 产品 | 是 | 所属产品线 | lmq / wuss |
| 迭代 | 否 | 所属迭代版本 | 如：v2.1.0 |
| TAPD_ID | 否 | TAPD需求ID | 数字ID |
| 创建日期 | 是 | 文档创建时间 | YYYY-MM-DD |
| 更新日期 | 是 | 文档更新时间 | YYYY-MM-DD |
| 版本号 | 是 | 文档版本 | V1.0, V1.1... |
| 依赖 | 否 | 依赖的其他模块 | 模块名称列表 |
| 关联接口 | 否 | 关联的API接口 | 接口名称列表 |

## 3. 章节结构规范

### 3.1 标准章节结构

```
一、业务背景
二、需求概述
三、功能需求
四、非功能需求
五、附录
```

### 3.2 各章节编写要求

#### 3.2.1 业务背景

**目的**：说明为什么要做这个功能，价值是什么。

**必含内容**：
- 业务现状与痛点
- 解决方案概述
- 核心价值（用列表说明）

**示例**：
```markdown
## 一、业务背景

包租婆是一个房产租赁管理系统，用于管理分散式和独栋公寓类型的房源。房屋标签管理模块用于统一维护和管理正式房源的属性标签。

核心价值包括：
- **房源标记**：为房源打上特色、配备、位置等多维度标签
- **视觉识别**：标签支持颜色和图标，提升房源信息的视觉识别度
- **灵活配置**：支持标签分类和标签项的动态维护
```

#### 3.2.2 需求概述

**目的**：概述功能定位、适用角色和范围边界。

**必含内容**：

| 章节 | 内容要求 |
|------|---------|
| 功能定位 | 用1-2句话说明功能是什么 |
| 适用角色 | 列出可使用该功能的角色 |
| 范围说明 | 明确功能边界，说明与其他模块的关系 |

**示例**：
```markdown
## 二、需求概述

### 2.1 功能定位

房屋标签管理模块主要实现对房屋标签分类和标签项的管理。

### 2.2 适用角色
- 系统管理员
- 后台运营人员

### 2.3 范围说明

本模块用于管理正式房源的标签，与字典管理模块的区别在于：
- **视觉属性**：标签支持颜色和图标
- **使用场景**：专用于房源标记
```

#### 3.2.3 功能需求

**目的**：详细描述每个功能点的页面、字段、交互规则。

**标准结构**：
```
### 3.1 列表页
  - 3.1.1 页面布局
  - 3.1.2 列表字段
  - 3.1.3 搜索筛选
  - 3.1.4 列表操作

### 3.2 新增/编辑页
  - 3.2.1 表单字段
  - 3.2.2 表单验证
  - 3.2.3 操作按钮

### 3.3 详情页
  - 3.3.1 页面布局
  - 3.3.2 信息展示
  - 3.3.3 操作功能

### 3.4 其他功能
  - 状态流转、导入导出等
```

#### 3.2.4 非功能需求

**必含内容**：

| 章节 | 内容要求 | 示例 |
|------|---------|------|
| 性能要求 | 页面加载、接口响应时间 | 页面加载时间：≤2秒 |
| 安全要求 | 权限、数据安全、审计 | 操作日志记录所有增删改操作 |
| 兼容性要求 | 浏览器、分辨率 | 支持Chrome、Firefox最新版本 |

#### 3.2.5 附录

**必含内容**：

| 章节 | 内容要求 |
|------|---------|
| 术语表 | 文档中涉及的专业术语解释 |
| 流程图 | 核心业务流程的可视化 |
| 版本历史 | 文档的变更记录 |

## 4. 表格设计规范

### 4.1 列表字段表

用于描述列表页的表格字段。

| 列名 | 说明 |
|------|------|
| 字段名称 | 字段的显示名称 |
| 说明 | 字段的含义和用途 |
| 数据来源 | 字段值的来源（用户输入/系统生成/系统计算） |

**示例**：
```markdown
| 字段名称 | 说明 | 数据来源 |
|---------|------|---------|
| 标签编码 | 标签的唯一标识 | 用户输入/系统预置 |
| 标签名称 | 标签的显示名称 | 用户输入 |
| 状态 | 启用/禁用 | 用户选择 |
| 创建时间 | 标签创建时间 | 系统自动记录 |
```

**默认排序规则**：

列表排序若无特殊业务要求，默认按**创建时间倒序**排列（最新数据在前）。

### 4.2 表单字段表

用于描述新增/编辑页的表单字段。

| 列名 | 说明 |
|------|------|
| 字段名称 | 字段的显示名称 |
| 是否必填 | 是/否/条件必填 |
| 字段类型 | 输入控件类型 |
| 说明 | 字段的含义和填写要求 |
| 数据来源 | 字段值的来源 |
| 示例 | 填写示例 |

**示例**：
```markdown
| 字段名称 | 是否必填 | 字段类型 | 说明 | 数据来源 | 示例 |
|---------|---------|---------|------|---------|------|
| 标签编码 | 是 | 文本输入 | 标签的唯一标识 | 用户输入 | feature |
| 标签名称 | 是 | 文本输入 | 标签的显示名称 | 用户输入 | 房源特色 |
| 默认颜色 | 否 | 颜色选择 | 该分类下标签的默认颜色 | 用户选择 | #67C23A |
```

### 4.3 操作按钮表

用于描述页面的操作按钮。

| 列名 | 说明 |
|------|------|
| 按钮名称 | 按钮显示的文字 |
| 按钮样式 | 按钮的视觉样式 |
| 功能描述 | 点击后执行的操作 |

**示例**：
```markdown
| 按钮名称 | 按钮样式 | 功能描述 |
|---------|---------|---------|
| 取消 | 白色按钮，灰色边框 | 关闭弹窗 |
| 确定 | 蓝色按钮（#409EFF） | 保存数据，关闭弹窗 |
```

### 4.4 预置数据表

用于说明系统初始化的默认数据。

**示例**：
```markdown
| 编码 | 名称 | 描述 | 归属 | 状态 |
|------|------|------|------|------|
| feature | 房源特色 | 用于标记房源的亮点特色 | 系统内置 | 启用 |
| equipment | 房源配备 | 用于标记房源的配套设施 | 系统内置 | 启用 |
```

### 4.5 业务主键命名规范

在需求文档中描述业务主键时，使用"编码"作为描述，不使用"ID"。

**原因**：
- "编码"具有业务含义，更易理解
- "ID"是技术术语，不适合出现在产品需求文档中
- 统一命名风格，保持文档一致性

**生成方式**：

| 方式 | 说明 | 适用场景 |
|------|------|---------|
| 手动填写 | 用户自行输入编码 | 小样本数据或需要遵循特定业务规则的场景 |
| 自动编码 | 通过编码中心自动获取 | 需要保证唯一性、无需人工干预的场景 |

**示例**：

❌ 错误写法：
```markdown
| 字段名称 | 说明 | 数据来源 |
|---------|------|---------|
| 任务ID | 任务的唯一标识 | 系统自动生成 |
```

✅ 正确写法：
```markdown
| 字段名称 | 说明 | 数据来源 |
|---------|------|---------|
| 任务编号 | 系统自动生成的唯一编号 | 系统自动生成 |
```

## 5. 流程图规范

### 5.1 流程图类型

| 类型 | 适用场景 |
|------|---------|
| 业务流程图 | 展示完整的业务操作流程 |
| 状态流转图 | 展示数据状态的变化过程 |
| 操作流程图 | 展示用户操作的步骤序列 |

### 5.2 文本流程图格式

推荐使用文本流程图，便于版本管理和修改：

```text
开始
  ↓
进入列表页
  ↓
点击"新增"按钮
  ↓
打开新增弹窗
  ↓
填写表单信息
  ↓
点击"确定"按钮
  ↓
表单验证
  ├─ 验证失败 → 显示错误提示 → 返回修改
  └─ 验证通过 → 保存数据
       ↓
     刷新列表
       ↓
     显示成功提示
       ↓
     结束
```

### 5.3 状态流转图格式

```text
                     [启用状态]
                         ↓
                   （点击禁用）
                         ↓
                    [禁用状态]
                         ↓
                   （点击启用）
                         ↓
                    [启用状态]
```

### 5.4 流程图符号约定

| 符号 | 含义 | 示例 |
|------|------|------|
| 开始/结束 | 流程的起点和终点 | `开始`、`结束` |
| 矩形 | 处理步骤 | `填写表单` |
| 菱形 | 判断条件 | `验证通过？` |
| ↓ | 流程方向 | 向下流转 |
| ├─ / └─ | 分支 | 条件分支 |
| → | 跳转 | 跳转到其他步骤 |

## 7. 常见问题与案例

### 7.1 常见问题

#### Q1: 业务背景应该写多长？

**建议**：控制在3-5句话，重点说明"为什么做"和"价值是什么"。避免冗长的背景铺垫。

#### Q2: 表单字段太多，如何组织？

**建议**：
- 按业务逻辑分组（如：基本信息、扩展信息）
- 使用子标题区分不同区域
- 必填字段在前，选填字段在后

#### Q3: 复杂的状态流转如何描述？

**建议**：
- 使用状态流转图可视化
- 配合文字说明每个状态变化的触发条件
- 列出所有可能的状态及其含义

#### Q4: 交互规则如何写清楚？

**建议**：
- 使用"当...时，执行..."的句式
- 分点列出不同条件下的行为
- 提供操作示例

### 8.4 新增菜单规范

新增功能模块时，需遵循以下规范：

1. **归属确认**：先确认新模块应归属的一级/二级菜单
2. **命名规范**：菜单名称应简洁明确，一般 2-4 个汉字
3. **层级控制**：菜单层级不超过三级，超过三级需拆分或合并
4. **PRD 对应**：PRD 文档目录命名格式为 `{一级菜单}-{二级菜单}[-{三级菜单}]`
