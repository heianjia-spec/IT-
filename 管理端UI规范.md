## 1. 设计原则

- **清晰优先**：信息层级明确，主任务路径不超过 3 步。
- **一致性**：同类页面使用同一布局与交互模式。
- **低认知负担**：减少装饰，突出数据与操作。
- **可扩展**：组件与间距可平滑支持更多字段与更大数据量。
- **中文优先**：所有界面文案、标签、按钮、提示、错误信息等均使用中文，确保用户理解无障碍。

## 2. 全局视觉与基础规范（Element 风格）

### 2.1 页面容器

- 页面背景：浅灰（如 `#f5f7fa`）
- 内容区卡片：白底，弱阴影或细边框
- 页面宽度：自适应，主内容区建议最小宽度 `1200px`
- 页面内统一使用 `Card + 表单 + 表格 + 分页` 结构

### 2.2 间距与节奏

- 页面区块上下间距：`16 / 20 / 24px`
- 表单项间距：`16px`
- 按钮组间距：`8px`
- 卡片内边距：`16px`（复杂区块可 `20/24px`）

### 2.3 字体层级

- 中文字体优先：推荐使用系统默认中文字体（如苹方、微软雅黑、思源黑体）
- 页面标题：`20px / 600`
- 区块标题：`16px / 600`
- 正文与表格：`14px`
- 辅助说明：`12px`

### 2.4 颜色语义

- 主色：Element 默认主色体系（如 `#409EFF`）
- 成功/警告/危险/信息：沿用 Element 语义色
- 文本：主文本、中级文本、次要文本分层使用
- 禁止新增非语义色作为状态色

## 3. 管理系统通用布局规范

### 3.1 经典后台布局（推荐默认）

- 顶部：系统名、全局搜索、通知、用户入口
- 左侧：一级/二级导航（可折叠）
- 右侧主区域：
  1. 面包屑
  2. 筛选区
  3. 操作区（主操作按钮：新增、导出等）
  4. 列表/详情主体

**适用**：大多数业务后台（用户、订单、权限、内容管理）

### 3.2 双栏工作台布局

- 左栏：筛选、目录、快捷入口
- 右栏：主内容（表格/详情）

**适用**：需要频繁切换分类与查看详情的页面

### 3.3 全宽表格布局

- 弱化卡片嵌套，主打数据密度
- 筛选在上，表格占满可用宽度

**适用**：运维监控、日志、审计等高密度数据场景

## 4. 列表页展示规范

### 4.1 标准结构

1. 查询区（筛选项 3~8 个，**默认最多显示两行**，超出增加"高级查询"按钮，通过 Drawer 抽屉实现）
2. 操作区（主操作按钮：新增、导出、导入等，位于查询区与表格之间）
3. 数据表格
4. 分页区（总数、页码、每页条数）

### 4.2 表格字段规则

- 关键字段左侧优先展示（名称、编号、状态）
- 操作列固定在右侧
- 时间字段统一格式（如 `YYYY-MM-DD HH:mm:ss`）
- 状态使用 `Tag` 呈现，颜色遵循语义色

### 4.3 列表交互建议

- 默认支持列排序（必要字段）
- 筛选条件支持”重置/查询”
- **查询操作交互规范**：
  - 输入查询条件后，点击”查询”按钮才执行查询操作
  - 不应在输入条件时实时触发查询（避免频繁请求和列表闪烁）
  - 点击”重置”按钮后，清空所有查询条件并恢复默认数据展示
  - 实现方式：使用 `ref` 存储表格数据，而非 `computed` 实时计算
  ```javascript
  // 推荐方式：点击查询后才过滤
  const tableData = ref([])
  const doSearch = () => {
    let result = rawData.value
    if (searchForm.name) result = result.filter(t => t.name.includes(searchForm.name))
    return result
  }
  const handleSearch = () => {
    pagination.current = 1
    tableData.value = doSearch()
  }
  const handleReset = () => {
    searchForm.name = ''
    tableData.value = rawData.value
    pagination.current = 1
  }
  // 初始化数据
  tableData.value = rawData.value
  ```
- 行内操作不超过 3 个，高频操作直出，其余放”更多”
- **”更多”按钮交互规范**：
  - 使用 `el-dropdown` 组件，设置 `trigger=”click”` 通过点击触发下拉菜单
  - 按钮 itself 不执行任何业务操作，仅作为下拉菜单的触发器
  - 按钮上添加 `@click.stop` 阻止事件冒泡，防止触发行点击事件
  - 下拉菜单项通过 `@click` 执行具体业务操作
  ```html
  <el-dropdown trigger=”click”>
    <el-button type=”primary” link size=”small” @click.stop>
      更多<el-icon class=”el-icon--right”><arrow-down /></el-icon>
    </el-button>
    <template #dropdown>
      <el-dropdown-menu>
        <el-dropdown-item @click=”handleAction(row)”>操作名称</el-dropdown-item>
      </el-dropdown-menu>
    </template>
  </el-dropdown>
  ```
- 空状态显示引导文案与新增入口

### 4.4 分页与数据加载

- 默认每页：`50`（可选 `20/50/100/200`）
- 总数大时必须分页，不建议无限滚动
- 所有查询操作应有加载态
- 中文

## 5. 详情页展示规范

### 5.1 标准结构（推荐）

1. 顶部信息区：标题、状态、主次操作（编辑、启停、删除）
2. 基本信息区：两列或三列描述（`Descriptions`）
3. 业务信息区：分卡片展示（可分组）
4. 关联数据区：子表格/日志/操作记录（`Tabs` 承载）

### 5.2 信息组织规则

- 先总览后细节：核心识别信息放最上方
- 不可编辑内容优先 `Descriptions`
- 可编辑内容使用“编辑态切换”或跳转编辑页
- 审计信息（创建人、创建时间、更新时间）置底或侧边

### 5.3 详情页操作规范

- 高风险操作（删除、停用）必须二次确认
- 状态切换动作需反馈（成功提示 + 状态即时更新）
- 返回路径明确（返回列表/返回上级）

### 5.4 详情页线框示例

```text

│ 返回列表                                                                     │
├──────────────────────────────────────────────────────────────────────────────┤
│ 标题：用户详情                     状态: [启用Tag]      [编辑] [停用] [删除]  │
├──────────────────────────────────────────────────────────────────────────────┤
│ 基本信息（Descriptions）                                                     │
│ 用户名: 张三      手机: 138****0000      邮箱: xxx@xx.com                    │
│ 角色: 管理员      部门: 运营中心        注册时间: 2026-01-10                 │
├──────────────────────────────────────────────────────────────────────────────┤
│ 业务信息卡片                                                                  │
│ [卡片A: 账户信息]   [卡片B: 安全信息]   [卡片C: 业务标记]                    │
├──────────────────────────────────────────────────────────────────────────────┤
│ Tabs: [关联订单] [操作日志] [变更记录]                                       │
```

### 5.5 详情页展示模式选型

详情页支持三种展示模式，根据场景灵活选用：

| 模式 | 组件 | 适用场景 | 屏幕占用 |
|------|------|----------|----------|
| **独立页面** | 页面跳转 | 信息量大、需独立 URL、有关联数据 | 全屏 |
| **弹窗对话框** | Dialog | 简单编辑、确认操作、字段少 | 400-800px |
| **侧边抽屉** | Drawer | 快速预览、保持列表上下文 | 400-720px |

#### 场景选型速查表

| 场景 | 推荐模式 |
|------|----------|
| 用户详情、订单详情、商品详情 | 独立页面 |
| 快速预览（点击行查看摘要） | Drawer |
| 简单编辑（修改状态、备注） | Dialog |
| 关联数据展示（订单明细、操作日志） | 独立页面 + Tabs |
| 审批、审核流程 | 独立页面 |
| 表格行内快速查看 | Popover / Tooltip |

#### 判断标准

| 维度 | 独立页面 | Dialog/Drawer |
|------|----------|---------------|
| **信息字段数** | > 15 个 | <= 15 个 |
| **操作复杂度** | 复杂编辑、多步骤 | 仅查看或简单编辑 |
| **关联数据** | 有子表格、日志、变更记录 | 无关联数据 |
| **URL 需求** | 需要独立 URL、可收藏/分享 | 无需独立入口 |
| **上下文** | 可以离开列表页 | 需保持列表上下文 |

### 5.6 独立页面详情（Page）

**适用条件（满足任意 2 条）**：
- 信息字段 > 15 个
- 需要展示关联数据（子表格、日志、变更记录）
- 需要复杂编辑操作（表单、多步骤）
- 业务需要独立 URL 入口

**结构规范**：见 5.1-5.4 章节

**交互规范**：
- 返回路径明确（返回列表/返回上级）
- 页面跳转后保持列表筛选状态
- 刷新页面不丢失数据

### 5.7 弹窗详情（Dialog）

**适用条件**：
- 信息字段 <= 10 个
- 操作简单（确认、状态切换、备注填写）
- 非频繁操作

**结构规范**：

```text
┌────────────────────────────────────────┐
│ 标题                              [×]  │
├────────────────────────────────────────┤
│                                        │
│ 内容区（表单/描述列表）                 │
│                                        │
├────────────────────────────────────────┤
│                    [取消]    [确定]    │
└────────────────────────────────────────┘
```

**尺寸建议**：
- 小型：400px（简单确认、备注）
- 中型：600px（表单编辑）
- 大型：800px（信息较多时）

**交互规范**：
- 点击遮罩层可关闭（可配置禁止）
- ESC 键可关闭
- 底部操作栏固定

### 5.8 抽屉详情（Drawer）

**适用条件**：
- 信息量中等（10-20 个字段）
- 需要保持列表上下文（对照查看）
- 快速预览 + 简单操作

**结构规范**：

```text
┌─────────────────────────────────────┐
│ 标题            状态  [编辑] [×]    │
├─────────────────────────────────────┤
│                                     │
│ 内容区（可滚动）                     │
│                                     │
│                                     │
├─────────────────────────────────────┤
│                    [取消]    [保存] │ ← 可选
└─────────────────────────────────────┘
```

**尺寸建议**：
- 默认：从右侧滑出，宽度 500px
- 大型：宽度 720px

**交互规范**：
- 从右侧滑入
- 点击遮罩层可关闭
- 可配置点击列表行打开

### 5.9 组合使用模式

#### 模式 A：列表 → Drawer 预览 → 页面详情

```text
列表页点击行 → Drawer 快速预览
         → 点击"查看详情" → 跳转详情页
```

**适用**：信息量大的业务（订单、商品、客户）

#### 模式 B：列表 → Dialog 编辑 → 保存刷新

```text
列表页点击"编辑" → Dialog 弹出表单 → 保存 → 列表刷新
```

**适用**：简单编辑场景（修改状态、添加备注）

## 6. 选择交互规范（小样本/大样本，单选/多选）

> 样本量定义：
> - **小样本**：通常 `<= 50` 条可选项
> - **大样本**：通常 `> 50` 条可选项

### 6.1 小样本单选

**推荐组件**：`Radio Group` / 下拉 `Select`（不可搜索也可）

- 选项少且互斥明显时优先 `Radio`
- 字段较短时可横向排列，较长时纵向排列
- 默认值建议明确给出（若业务允许）

> ⚠️ **注意**：`el-radio-group` 内的 `el-radio` 必须使用 `label` 属性绑定值，详见 [8.1 Radio Group 值绑定规范](#81-radio-group-值绑定规范)

### 6.2 小样本多选

**推荐组件**：`Checkbox Group`

- 适合标签、功能开关组合、少量权限点
- 支持”全选/取消全选”
- 多选结果建议就近显示已选数量

> ⚠️ **注意**：`el-checkbox-group` 内的 `el-checkbox` 必须使用 `label` 属性绑定值，详见 [8.2 Checkbox Group 值绑定规范](#82-checkbox-group-值绑定规范)

### 6.3 大样本单选

**推荐组件**：`Select（filterable）` / `Remote Select` / `弹窗选择器`

- 必须支持搜索（本地或远程）
- 选项应展示关键辅助信息（如编号、类型）
- 若需要看更多字段，使用“弹窗表格单选”更清晰

### 6.4 大样本多选

**推荐组件**：`Select（multiple + filterable）`、`Transfer`、`弹窗表格多选`

- 数据量大且需跨页选择：优先”弹窗表格多选 + 已选池”
- 数据结构稳定、左右对照明显：可用 `Transfer`
- 多选结果必须可回显、可快速移除
- 跨页批量选择时需提示”已选 N 项（跨页）”

### 6.5 多级选择（单选）

**推荐组件**：`Cascader（级联选择）`、`Tree Select（树选择单选）`、`弹窗树单选`

**适用场景**：组织架构选择、地区选择、多级分类选择、菜单权限选择

- **层级少（2-3 级）且节点少**：优先 `Cascader`，交互简洁直观
- **层级深或节点多**：使用 `Tree Select`，支持搜索展开
- **需要查看更多节点信息**：使用”弹窗树单选”，可展示节点详情

**交互规则**：
- 必须支持搜索（按节点名称/编码）
- 显示完整层级路径（如：技术部 / 前端组 / UI 小组）
- 叶子节点才可选中，或按业务允许选中任意层级
- 选中后显示完整路径，hover 显示全路径 tooltip

### 6.6 多级选择（多选）

**推荐组件**：`Tree（Checkbox 模式）`、`Tree Select（multiple）`、`弹窗树多选 + 已选池`

**适用场景**：批量分配部门、多级权限配置、批量分类选择

- **层级少且节点少**：使用 `Tree Select（multiple）`，下拉框内嵌树
- **层级深或节点多**：使用”弹窗树多选 + 已选池”，支持跨层级批量选择
- **需要父子联动**：父子节点选中状态关联（选父自动选子，选子自动选父）

**交互规则**：
- 支持节点搜索并自动展开匹配路径
- 支持全选/取消全选（当前层级或全部）
- 必须展示”已选 N 个节点”，支持快速定位和移除
- 父子联动模式需提供”严格模式”选项（父子独立选中）
- 大批量选择时使用已选池，支持按层级分组展示已选项

### 6.7 选择器通用规则

- 占位文案明确（如“请选择角色”）
- 禁用态说明原因（如权限不足、前置条件未满足）
- 异步加载场景提供 loading 与空结果提示
- 已选值与提交值（label/value）保持映射一致

## 7. 场景选型速查表

| 场景 | 推荐方式 | 不推荐方式 |
|---|---|---|
| 5~8 个固定互斥项 | Radio | 下拉 Select |
| 10~30 个固定多选项 | Checkbox Group | 弹窗选择 |
| 100+ 单选并可搜索 | Filterable Select / 弹窗单选 | Radio 列表 |
| 100+ 多选且跨页操作 | 弹窗表格多选 + 已选池 | 普通多选下拉 |
| 2-3 级层级单选（节点少） | Cascader 级联选择 | 多个下拉联动 |
| 深层级单选（节点多） | Tree Select（可搜索） | 普通下拉 |
| 2-3 级层级多选（节点少） | Tree Select（multiple） | 多个 Checkbox |
| 深层级多选（节点多） | 弹窗树多选 + 已选池 | 普通树多选 |
| 组织架构/地区选择 | 弹窗树（支持搜索+已选池） | 扁平下拉 / 多个下拉 |
| 权限配置（父子联动） | Tree（Checkbox 模式 + 联动） | 独立 Checkbox |

## 9. Element Plus 使用注意事项

本章节汇总 Element Plus 组件使用中容易出错的关键规范。

### 9.1 Radio Group 值绑定规范

在 `el-radio-group` 内的 `el-radio` **必须使用 `label` 属性**绑定值，而非 `value` 属性。

**错误用法**（会导致选项联动异常）：
```html
<el-radio-group v-model=”status”>
  <el-radio :value=”1”>启用</el-radio>  <!-- ❌ 错误 -->
  <el-radio :value=”0”>禁用</el-radio>
</el-radio-group>
```

**正确用法**：
```html
<el-radio-group v-model=”status”>
  <el-radio :label=”1”>启用</el-radio>  <!-- ✅ 正确 -->
  <el-radio :label=”0”>禁用</el-radio>
</el-radio-group>
```

**原因**：Element Plus 的 `el-radio-group` 组件内部通过 `label` 属性来识别和匹配选中值。使用 `value` 属性会导致所有 radio 共享同一个引用，从而出现选项异常。

### 9.2 Checkbox Group 值绑定规范

在 `el-checkbox-group` 内的 `el-checkbox` **必须使用 `label` 属性**绑定值，而非 `value` 属性。

**错误用法**（会导致”选一个全选”的异常）：
```html
<el-checkbox-group v-model=”selectedDays”>
  <el-checkbox :value=”1”>周一</el-checkbox>  <!-- ❌ 错误 -->
  <el-checkbox :value=”2”>周二</el-checkbox>
</el-checkbox-group>
```

**正确用法**：
```html
<el-checkbox-group v-model=”selectedDays”>
  <el-checkbox :label=”1”>周一</el-checkbox>  <!-- ✅ 正确 -->
  <el-checkbox :label=”2”>周二</el-checkbox>
</el-checkbox-group>
```

**原因**：Element Plus 的 `el-checkbox-group` 组件内部通过 `label` 属性来识别和匹配选中值。使用 `value` 属性会导致所有 checkbox 共享同一个引用，从而出现”选一个全选”的异常行为。

## 11. 交付建议（给设计与前端）

### 11.1 开发前准备

- 每个页面先确定”布局模板”（经典后台/双栏/全宽）
- 每个选择器先判断”样本量 + 单/多选”，再选组件
- 多级选择先判断”层级深度 + 节点数量”，再选 Cascader 或 Tree
- 所有列表页必须统一：查询区 + 表格 + 分页
- 所有详情页必须统一：总览 + 分组信息 + 关联记录
- **同一个功能菜单生成的 UI 界面，所有元素必须包含在同一个 HTML 脚本中**（包括样式、脚本、交互逻辑），确保功能独立可运行

### 11.2 最小落地标准（MVP）

- 仅保留必要筛选项与高频操作
- 表格字段控制在”核心字段 + 状态 + 操作”
- 详情页先展示只读信息，编辑流程独立
- 选择控件严格按样本量选型，避免过度设计
